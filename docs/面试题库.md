## 一、数据库 · 索引与执行计划

### Q1【理论】`EXPLAIN ANALYZE` 输出里的关键字段都是什么意思？

| 字段 | 含义 |
|---|---|
| 节点类型(Seq Scan / Index Scan / Bitmap Index Scan) | 数据库取数据的方式 |
| `cost=启动..总` | 规划器**估算**的工作量，相对值（非毫秒），用于比较不同执行计划 |
| `rows=`(括号第一组) | 规划器**估算**返回行数 |
| `width=` | 每行估算字节数 |
| `actual time=首行..末行` | **真实**耗时(ms) |
| `rows=`(actual) | **真实**返回行数 |
| `loops=` | 节点执行次数；actual time 是单次的，总耗时 ×loops |
| `Filter:` | 读出行**之后**再过滤的条件（低效信号） |
| `Rows Removed by Filter:` | 读出又丢弃的行数，数字大=浪费=该加索引 |
| `Index Cond:` | 下推到**索引层**过滤的条件（高效信号） |
| `Execution Time` | 真实总耗时（小表上仅供参考） |

**两个优化杀手锏：**
1. 条件落在 `Filter`（读完再筛）=没用上索引；目标是让它变成 `Index Cond`。
2. 估算 `rows` 与实际 `rows` 差很多 → 统计信息过期 → 执行 `ANALYZE 表名;`。

### Q2【理论】为什么加了索引，查询时间却几乎没变？

因为数据量小（如 10 万行）整表在内存里，墙上时间被噪音淹没。**要看"工作量"而非时间**：
- `cost` 从 2037 → 41（约 50 倍）
- `Rows Removed by Filter: 99986`（翻遍全表）→ `Heap Blocks: exact=14`（只碰 14 块）

数据涨到百万行时，这个工作量差距才会变成"2ms vs 2s"。**`cost` 是判断索引是否有效的硬指标。**

### Q3【理论】什么是最左匹配原则？只给组合索引的非最左列会怎样？

组合索引 `(A, B)` 按"先 A 再 B"排序，像电话簿按"姓+名"排。
- 给 `A` 或 `A+B` → 高效命中（cost 低）
- 只给 `B` → **用不上（或退化为低效全索引扫描）**

**关键纠正**：最左匹配不是"能不能查出结果"（结果永远正确），而是"能不能**高效**用上索引"。

实测（组合索引 `(status, created_at)`，PostgreSQL）：

| 查询 | 索引扫描 cost | 说明 |
|---|---|---|
| 只 `status` | ≈281 | 命中最左列，高效 |
| 只 `created_at` | ≈1114 | 跳过最左列，扫整棵索引，最贵 |
| `status`+`created_at` | ≈9.6 | 从最左连续命中，最优 |

**差异**：MySQL 中"只给非最左列"通常直接放弃索引走全表；PostgreSQL 可能仍扫整棵索引（全索引扫描），但代价同样很高。

### Q4【理论】索引到底什么时候该建？判断标准是什么？

核心是**选择性(Selectivity)**——条件能筛掉多少行，不只是数据量。

| 查询 | 返回占比 | 选择性 | 索引效果 |
|---|---|---|---|
| `user_id=5000` | 0.01% | 高 | 🔥 有效(cost 2037→41) |
| `status='paid'` | 25% | 低 | 😐 几乎没用 |

- **该建**：高频出现在 WHERE/JOIN/ORDER BY 的**高选择性**列（主键、邮箱、user_id、订单号）；读多写少的表。
- **别建**：小表（几千行直接全表扫）；低选择性列（status、性别、布尔）；很少查的列；写密集表别堆索引（每次写都要维护所有索引）。
- 经验值：查出占比超过约 20~30% 时，数据库往往宁愿全表扫描。

### Q5【场景】管理后台表格顶部有很多筛选条件，可任意组合、也可单独查，索引怎么建？有必要吗？

1. **先问有没有必要**：后台表常就几千~几万行，全表扫也就几毫秒，别凭空加；等 `EXPLAIN` 真发现慢查询再加。
2. **任意组合 → 多个单列索引 + 数据库自动合并**：PostgreSQL 用 `BitmapAnd`/`BitmapOr` 在查询时临时合并多个单列索引。给少数几个**高选择性、高频**的列各建单列索引，让数据库自己组合，而不是猜组合顺序建一堆组合索引。
3. **会单独查的列**（哪怕它在某组合索引末尾）→ 给它**单独**建单列索引（组合索引 `(a,b,c)` 帮不了"只查 c"）。
4. **组合索引只为高频固定组合建**，典型两种：
   - 等值+范围：`WHERE device_id=? AND recorded_at>?` → `(device_id, recorded_at)`
   - 筛选+排序分页：`WHERE status=? ORDER BY created_at DESC LIMIT n` → `(status, created_at)`（索引顺便完成排序，避免额外 Sort）

**一句话**：任意组合靠"几个高选择性单列索引 + BitmapAnd 合并"；只为高频固定组合（尤其等值+范围、筛选+排序）建组合索引；低选择性列和小表不建；一切以 `EXPLAIN` 为准。

### Q6【理论】SQL 单表查询的 WHERE 可以不写表前缀，为什么业务代码都带表名？

- 单表查询：`WHERE user_id=5000` 无歧义，可省前缀。
- 多表 JOIN：`id`、`created_at` 等列名可能多表都有，不加前缀会报 `ambiguous column`。
- 业务查询几乎都涉及 JOIN，带前缀既防歧义又让字段归属一目了然，是好习惯。

### Q7【理论/场景】SQLAlchemy 里怎么给表定义索引？组合索引列顺序怎么定？

在模型类里用 `__table_args__`（元组）+ `Index()`：

```python
from sqlalchemy import Index

class DeviceTelemetry(BaseModel):
    __tablename__ = "device_telemetry"
    __table_args__ = (
        Index("idx_device_telemetry_did_metric_time",
              "device_id", "metric", "recorded_at"),
    )
```

要点：
- `__table_args__` 是**元组**，单个索引也要写**结尾逗号**，否则不是元组。
- 列顺序黄金法则：**等值条件列在前**（device_id、metric），**范围/排序列在后**（recorded_at）。
- 好处：支撑 `WHERE device_id=? AND metric=?` 的同时，`recorded_at` 可直接满足 `ORDER BY recorded_at`（省去额外 Sort）；并白送 `(device_id)`、`(device_id, metric)` 两种前缀查询能力（最左匹配）。
- 单列索引则用 `mapped_column(..., index=True)`。

### Q8【理论】什么是「回表」？什么是「覆盖索引 / Index Only Scan」？

- **回表（Heap Fetch）**：B-tree 索引的叶子节点只存「索引列 + 指向数据行的指针」。如果 `SELECT` 要的列不在索引里，数据库得拿指针再去**表（堆 heap）**把整行捞出来——这一步叫回表。行越多越分散越贵。
- **覆盖索引（covering index）**：一条查询用到的**所有列**（`SELECT` + `WHERE` + `ORDER BY`）都能在索引里找到，数据库就不回表，直接从索引返回——执行计划里显示为 **Index Only Scan**，`Heap Fetches: 0`。称这个索引「覆盖」了这条查询。

实测（`device_telemetry` 30 万行，组合索引 `(device_id, metric, recorded_at)`，PostgreSQL 17）：

| 查询 | 扫描方式 | 回表 | Buffers(碰到的页) | cost |
|---|---|---|---|---|
| `SELECT value` | Bitmap Heap Scan | Heap Blocks=218 | **225** | ≈811 |
| `SELECT recorded_at` | **Index Only Scan** | Heap Fetches=0 | **5** | ≈14.7 |

同样的 WHERE，只因为「取的列在不在索引里」，碰到的页数差约 **45 倍**、cost 差约 **55 倍**。

### Q9【理论/场景】PostgreSQL 怎么把一个「要回表的查询」改成覆盖索引？`INCLUDE` 和直接写进索引键有什么区别？

用 `INCLUDE` 把「只想顺带返回、不参与查找排序」的列塞进索引叶子：

```sql
CREATE INDEX idx_dt_covering ON device_telemetry
    (device_id, metric, recorded_at) INCLUDE (value);
```

实测：原本 `SELECT value` 要回表（cost≈811、225 页），加了 `INCLUDE(value)` 后变成 Index Only Scan、`Heap Fetches: 0`、cost≈14.5、6 页。

**`INCLUDE(value)` vs 写进键 `(..., recorded_at, value)`：**
- 键列负责**定位 + 排序 + 最左匹配**；`value` 不参与查找和排序，只想「免回表」。
- 写进键会让索引更大、维护更贵，还可能干扰排序语义。
- `INCLUDE` 只在叶子附带存一份，职责清晰、代价更小——这正是 `INCLUDE` 的用途。

**一句话**：高频查询固定只取某几列时，把这几列做成覆盖索引（缺的列用 `INCLUDE` 补），让它走 Index Only Scan。

### Q10【场景】Index Only Scan 的两个隐藏要点：`Heap Fetches` 不为 0 / 额外的 Sort 不见了

1. **`Heap Fetches` 不为 0 = 假的 Index Only Scan**：Index Only Scan 靠**可见性图谱（visibility map）**判断「这一页的行是否全部可见」，只有可见才敢不回表。刚灌完/刚大量写入、还没 `VACUUM` 时图谱是空的，仍会大量回表（`Heap Fetches` 很大）。**先 `VACUUM ANALYZE 表名;`** 才能看到真正的 Index Only Scan。读多写少的表效果最稳。
2. **覆盖索引有时还白送排序**：实测里 `SELECT value`（回表）走 Bitmap Heap Scan，它按数据页物理顺序捞行、丢了索引有序性，所以 `ORDER BY recorded_at` 要额外加一个 **Sort** 节点；而 Index Only Scan 顺着 B-tree 走，天然按 `recorded_at` 有序，`ORDER BY` 直接满足、**没有 Sort**。
   → 覆盖索引省的不只是「回表的页」，有时还省掉「额外的排序」，双重收益。

**附带工程结论**：别习惯性 `SELECT *`。`SELECT *` 要的列（`created_at`/`is_deleted`…）覆盖索引装不下，必然回表；只取需要的列，才有机会吃到 Index Only Scan。

## 二、数据库 · 事务与隔离级别

### Q11【理论/场景】事务边界应该放在哪一层？为什么 Repository 不能自己 commit？

**一句话**：事务边界 = 一个「业务操作」的边界，所以放在 **Service 层**（一个业务方法）开启和提交；**Repository 只做单表 `add`/`delete`/改属性，自己不 commit、不 refresh**。

- **为什么 repo 不能 commit**：一个业务操作可能含多次写（转账=A 扣钱 + B 加钱）。若每个 repo 方法各自 commit，就成了多个独立事务；第二次写失败时，第一次已落库、**无法整体回滚**，数据变脏。
- **多分支 / 早返回怎么统一提交**：用 `with db.begin():` 把整段业务包起来——提交/回滚绑定到「离开代码块」，不管从哪个分支 `return`（→ commit）还是 `raise`（→ rollback）都自动做对，不用在分支里写 commit。
- **refresh 归谁**：通常不需要。Service commit 时会先 flush，自增 `id` 自动写回对象；`expire_on_commit=True` 默认会在 commit 后访问属性时自动重查（请求级 session 还开着）。只有「数据库端生成的列」（`server_default`/触发器/计算列）才需在 Service commit 后 refresh。

| | add/delete/改属性 | commit | refresh |
|---|---|---|---|
| Repository | ✅ | ❌ | ❌ |
| Service | ❌ | ✅（`with db.begin()`） | 仅当有 DB 端生成列，commit 后做 |

**注意点**：① 边界放 Service 不放 Repository；② repo 不 commit；③ 事务要短，发邮件/调第三方/慢循环放 commit 之后；④ 异常必须回滚（用 `with` 自动化）；⑤ 纯查询不用显式事务。

### Q12【理论】什么是隔离级别？三种并发异常和四个级别分别是什么？PostgreSQL/MySQL 默认是哪个？

**一句话**：隔离级别 = 多个事务并发时，一个事务能看见另一个「未提交」改动的程度；越松越快越乱，越严越安全越慢。

**三种并发异常：**
| 异常 | 含义 |
|---|---|
| 脏读 | 读到别的事务**未提交**的改动（对方回滚后即成幻觉） |
| 不可重复读 | 同一事务内**两次读同一行**结果不同（中间别人改了该行并提交，针对 UPDATE） |
| 幻读 | 同一事务内**两次同条件范围查**行数变了（中间别人插/删符合条件的行，针对 INSERT/DELETE） |

**四个级别（松→严，越严禁止的异常越多）：**
| 级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| 读未提交 Read Uncommitted | 可能 | 可能 | 可能 |
| 读已提交 Read Committed | 禁止 | 可能 | 可能 |
| 可重复读 Repeatable Read | 禁止 | 禁止 | 可能* |
| 串行化 Serializable | 禁止 | 禁止 | 禁止 |

**默认值（常考）：**
- **PostgreSQL 默认 = 读已提交（Read Committed）**；且 PG 的「可重复读」基于快照(MVCC)实现，连幻读也基本挡住（上表 `*`）。
- **MySQL(InnoDB) 默认 = 可重复读（Repeatable Read）**。
- 跨库要点：同样代码在 PG 与 MySQL 下默认隔离级别不同，行为可能不一致。

**SQLAlchemy 设置**：`create_engine(url, isolation_level="REPEATABLE READ")`（engine 级）或 `engine.connect().execution_options(isolation_level="SERIALIZABLE")`（连接级，设 `"AUTOCOMMIT"` 也走这里）。

### Q13【理论/场景】数据库锁：解决什么问题？行锁/表锁、读锁/写锁、死锁分别是什么？SQLAlchemy 怎么主动加锁？

**一句话**：锁 = 给数据加「使用中」标记，防止多个事务同时改同一份数据时互相覆盖、读到半成品。

**为什么需要——丢失更新（lost update）**：A、B 同时读 balance=100，A 写回 110、B 写回 120，A 的 +10 被覆盖消失（正确应是 130）。锁让对同一行的修改排队。

**两个维度：**
- **读锁(S)/写锁(X)**：读锁可多事务共享，写锁同时只一个；规则=**读读不冲突，读写/写写冲突**。
- **行锁/表锁**：行锁只锁动的行、并发好（PG / MySQL InnoDB 默认）；表锁锁整张表、并发差（DDL/批量）。粒度越细并发越好、管理开销越大。

**死锁**：两事务各持一锁又互等对方的锁（A 持行1等行2，B 持行2等行1）→ 互相干等。数据库**自动检测并回滚其中一个**，应用需捕获错误重试。**预防**：所有事务**按相同顺序加锁**（如都先小 id 后大 id），避免成环。

**SQLAlchemy 主动加锁：**
```python
# 悲观锁：读的同时加排他锁，别人想改/想 FOR UPDATE 得排队
acct = session.query(Account).filter_by(id=1).with_for_update().first()
acct.balance += 10
session.commit()   # 提交后释放
```
- **悲观锁**（`SELECT ... FOR UPDATE`）：假设一定冲突，先锁；适合写多。
- **乐观锁**（`version` 列 + `WHERE version=旧值`，失败重试）：适合读多写少。

**和隔离级别的关系**：隔离级别很大程度靠锁实现——级别越高，锁越多/持有越久/冲突越频繁（串行化甚至频繁回滚重试），并发越慢。这就是「安全 vs 性能」权衡的底层机制。
